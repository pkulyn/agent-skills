# BATCH 批量处理工作流

## 触发条件

用户请求处理多个文件/文件夹，或提到"batch"、"批量"、"批量处理"

## 核心特性

- 幂等性保证：通过registry.json去重
- 增量更新：仅处理新增/修改的文件
- 错误隔离：单个文件失败不影响整体
- 进度跟踪：分批汇报
- 可回滚：通过快照机制支持一键撤销
- 断点续传：任务中断后可从上次进度继续

## 批量任务配置选项

| 参数 | 可选值 | 默认值 | 说明 |
|------|--------|--------|------|
| source | 文件夹路径 / 文件列表 / 过滤规则 | 必填 | 要处理的源文件位置 |
| auto_create_entities | true / false / "ask" | "ask" | 是否自动创建实体页面 |
| conflict_strategy | "merge" / "overwrite" / "skip" / "ask" | "merge" | 实体/主题冲突时的处理策略 |
| require_approval | true / false | false | 是否需要人工审核每个文件的处理结果 |
| incremental | true / false | true | 是否仅处理新增/修改的文件 |
| batch_size | 5-50 | 10 | 每批处理的文件数量 |

## 执行步骤

### 第1步：任务初始化

1. 验证源文件路径/列表有效性
2. 读取`Wiki/.state/registry.json`，获取已处理文件记录
3. 扫描所有符合条件的.md/.txt/.html文件，计算MD5哈希
4. 分类文件：新增 / 需更新 / 跳过 / 源已删除
5. 生成任务ID（格式：`batch-YYYYMMDD-HHMMSS`）
6. 创建快照：对本次任务将修改/覆盖的所有已有Wiki页面，复制原内容到`Wiki/.snapshots/{task-id}/`
7. 创建快照清单`Wiki/.snapshots/{task-id}/manifest.json`
8. **快照创建校验**（必执行）：验证manifest.json和snapshotFile均存在，否则暂停报错
9. 在`Wiki/log.md`中记录BATCH任务开始

### 第2步：分批执行

- 每批处理`batch_size`个文件（默认10个）
- 对每个文件执行标准INGEST流程
- 每批完成后更新registry.json、log.md、汇报进度

进度汇报格式：
- 单文件级：`✅/⚠️/❌ {文件名} → {生成页面数量}个页面 | {耗时}秒`
- 批次级：统计数据（已处理/成功/失败/跳过/累计生成/速率/预估剩余）

### 第3步：冲突处理

| 策略 | 执行规则 | 适用场景 |
|------|----------|----------|
| **merge**（默认） | 实体：追加补充视角章节；主题：追加新条目；摘要：哈希变化时直接重新生成 | 日常批量处理 |
| **overwrite** | 直接替换整个页面（必须先快照） | 旧信息完全过时 |
| **skip** | 保留原有内容不变 | 留待人工判断 |
| **ask** | 暂停，向用户展示冲突详情 | 高价值页面需人工决策 |

### 第4步：错误处理

- 单文件失败：记录错误，标记为"失败"，继续下一文件
- 可重试错误（`retryable: true`）：自动重试1次
- 不可重试错误（文件为空、格式损坏）：不重试

### 第5步：任务完成

1. 生成批量处理报告到`Wiki/reports/batch-{task-id}.md`
2. **报告校验**（必执行）：跳过列表校验、统计一致性校验、生成页面存在性校验、禁止虚构
3. 更新index.md统计（Glob精确计算）
4. 在log.md记录完成
5. 向用户展示结果摘要
6. 询问是否回滚

### 第6步：回滚（可选）

1. 读取manifest.json获取受影响页面
2. create页面→删除；update页面→从快照恢复
3. 更新registry.json、index.md、log.md
4. 删除快照目录
5. 确认回滚完成

### 断点续传

任务中断后重新启动时：读取registry.json识别已处理文件 → 自动跳过 → 询问用户处理失败文件
